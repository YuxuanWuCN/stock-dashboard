# 家庭股票自动看板 · 项目背书（规格文档）

> **文档用途**：本文档是给两位 AI 协作者的开发规格书。
> - **DeepSeek** 负责：数据抓取、指标计算、GitHub Actions 自动化、整体串联（第 3~7 章）。
> - **Gemini** 负责：网页可视化 / 前端呈现（第 8 章，须严格遵守第 6 章的数据契约）。
>
> **给两位 AI 的最高原则**：本项目服务于两位不方便盯盘的中老年用户，最终产物是一个"手机打开就能看懂"的看板。**简单、稳定、看得懂** 优先于任何炫技。凡本文档已规定的字段名、文件名、目录结构、接口约定，**一律不得擅自更改**，否则两边产物会对不上。文档未规定处，选择最简单、最少依赖的做法。

---

## 1. 项目背景与目标

### 1.1 背景
一对中老年夫妇有炒股需求，但没有足够时间和精力盯盘。子女希望做一个自动化程序，把他们关注的股票 / 基金数据，每天收盘后自动整理成直观的 K 线看板，让父母用**手机**随时打开一个固定网址就能查看，无需任何操作。

### 1.2 目标（一句话）
**每个交易日晚上自动抓取自选股票/基金数据，生成含 K 线、成交量、均线和涨跌摘要的可视化网页，自动发布到公网固定网址，父母手机随时可看；抓取失败自动邮件提醒维护者。**

### 1.3 核心设计约束
1. **零日常维护**：配置一次后，长期自动运行，父母和维护者平时都不需要动手。
2. **不依赖家用电脑**：整个流程运行在云端（GitHub Actions），家里电脑关机不影响。
3. **零服务器成本**：全部使用 GitHub 免费额度（Actions + Pages）。
4. **面向中老年用户的可读性**：字大、色明、涨红跌绿（A股习惯）、一屏看懂重点。
5. **数据真实**：只使用真实公开数据源，严禁编造、伪造、填充任何行情数值。

## 3. 总体技术方案

### 3.1 架构总览

```

│     · 产出标准化数据文件到 docs/data/（见第 6 章数据契约）                 │
│         │                                                                 │
│         ▼                                                                 │
│  ③ 前端页面（Gemini 编写，位于 docs/）                                     │
│     · 读取 docs/data/ 下的 JSON                                           │
│     · 用 ECharts 渲染 K 线看板                                            │
│         │                                                                 │
│         ▼                                                                 │
│  ④ GitHub Pages 自动发布 docs/ 目录 → 得到公网固定网址                     │
│         │                                                                 │

父母：手机浏览器打开固定网址（可存为书签 / 加到主屏）→ 看最新看板
```

### 3.2 为什么这么选（给两位 AI 的选型依据，勿随意替换）

- **数据源：akshare**。免费、开源、覆盖 A 股与场内基金、支持前复权，接口稳定且社区活跃。核心接口经确认可用（详见第 5 章）。
- **定时与运行：GitHub Actions**。免费、云端常驻、自带 cron 定时、失败自动邮件通知，无需自建服务器。
- **发布：GitHub Pages**。免费、与仓库同源、发布 `docs/` 目录即得公网网址，天然适配"生成静态文件→发布"的流程。
- **图表库：ECharts（推荐 5.x）**。国产、文档中文、K 线（candlestick）支持完善、移动端友好、可用 CDN 引入零构建。**前端不要引入 React/Vue/打包工具**，保持纯静态 HTML + JS，降低维护复杂度。

### 3.3 关键设计决策：数据与前端解耦
- Python 脚本（DeepSeek）**只负责产出数据文件**，不负责画图。
- 前端（Gemini）**只负责读取数据文件画图**，不关心数据怎么来的。
- 两者之间靠**第 6 章的数据契约（JSON 结构）**对接。**这是本项目最重要的接口，双方必须严格遵守。**

### 4.1 自选股列表 `watchlist.csv`（用户可自行编辑的唯一文件）

设计目标：让不懂代码的人也能改。用 CSV，Excel / 记事本都能打开编辑。

**列定义（表头固定，逗号分隔，UTF-8 编码）：**

| 列名 | 含义 | 示例 | 必填 |
|---|---|---|---|
| `code` | 标的代码（6 位数字，不带交易所后缀） | `600519` | 是 |
| `name` | 显示名称（用户自定义，中文） | `贵州茅台` | 是 |
| `type` | 类型：`stock`=股票，`etf`=场内基金/ETF | `stock` | 是 |

**示例内容：**
```csv
code,name,type
600519,贵州茅台,stock
000001,平安银行,stock
300750,宁德时代,stock
510300,沪深300ETF,etf
159915,创业板ETF,etf
```

**DeepSeek 须实现的健壮性要求：**
- 忽略空行和以 `#` 开头的注释行。
- 代码列做校验：非 6 位数字的行跳过并在日志中警告。
- `type` 非 `stock`/`etf` 的按 `stock` 处理并警告。
- 允许列表里有 10~30 行；超出也应能正常工作，只是运行时间变长。
- 文件不存在或为空时，报明确错误并触发失败邮件。

## 5. 数据抓取规格（DeepSeek 负责）

### 5.1 数据源与接口（经确认可用，2026-07 核对）

**股票（A 股，type=stock）——使用 akshare `stock_zh_a_hist`：**
```python
import akshare as ak
df = ak.stock_zh_a_hist(
    symbol="600519",         # 6位代码，A股无需后缀
    period="daily",          # 日线
    start_date="20250718",   # 见 5.2 计算方式
    end_date="20260718",
    adjust="qfq",            # 前复权（本项目固定用 qfq）
)
```
返回列（中文列名，DeepSeek 须映射到第 6 章英文字段）：`日期, 股票代码, 开盘, 收盘, 最高, 最低, 成交量, 成交额, 振幅, 涨跌幅, 涨跌额, 换手率`。

**场内基金 / ETF（type=etf）——使用 akshare `fund_etf_hist_em`：**
```python
df = ak.fund_etf_hist_em(
    symbol="510300",
    period="daily",
    start_date="20250718",
    end_date="20260718",
    adjust="qfq",
)
```
返回列与股票接口基本一致（`日期, 开盘, 收盘, 最高, 最低, 成交量, 成交额, 振幅, 涨跌幅, 涨跌额, 换手率`）。

> **备用源约定**：若上述东财接口某日不可用，DeepSeek 可实现降级重试（如新浪 `stock_zh_a_daily` / 腾讯 `stock_zh_a_hist_tx`，同样支持 `adjust="qfq"`），但**产出的数据字段结构必须与第 6 章完全一致**，前端无感知。降级仍失败则按第 7 章走失败流程。

### 5.2 抓取参数
- `end_date` = 运行当天日期。
- `start_date` = 运行当天往前推 **400 个自然日**（保证覆盖"最近约 250 个交易日 ≈ 1 年"，多抓一些以便算 MA60 时首日也有均线，前端再截取展示最近 1 年）。
- `adjust` 固定 `"qfq"`。
- `period` 固定 `"daily"`。

### 5.3 衍生指标计算（在 Python 端算好，前端不算）
对每只标的，基于**收盘价**计算并写入 K 线数据：
- **MA5、MA10、MA20、MA60**：N 日收盘价简单移动平均，保留 2 位小数。数据不足 N 天的位置填 `null`（前端不连线）。
- **当日涨跌幅**：优先用接口返回的"涨跌幅"字段；若无则用 `(今收 - 昨收)/昨收 × 100`，保留 2 位小数。
- **当日涨跌额**：`今收 - 昨收`，保留 2 位小数。

### 5.4 抓取工程要求（重要，直接影响稳定性）
1. **限流保护**：逐只串行抓取，每只之间 `sleep` 1~2 秒，避免东财封 IP。**不要并发猛抓。**
2. **单只失败隔离**：某一只抓取失败（超时/异常/空数据）时，**记录该只失败，跳过并继续抓其他只**，不要让一只失败拖垮整批。
3. **日期排序**：结果**必须按日期升序排列**（老→新）。切记不要对日期做字符串排序，须转成日期类型排序，避免 "2026-12" 排到 "2026-02" 前面的经典错误。
4. **数据清洗与校验**（数据边界必须校验）：
   - 开/高/低/收必须为正数；出现 ≤0 或缺失的行，剔除该行并警告（东财个别复权数据存在负值异常）。
   - 校验 高 ≥ max(开,收)、低 ≤ min(开,收)，异常行警告。
   - 成交量非负。
   - 单只标的最终有效行数 < 20 时，视为该只抓取失败（数据太少无意义）。
5. **原子写入**：先写临时文件再重命名，避免写一半被中断导致前端读到坏 JSON。
6. **保留旧数据**：若整批全部失败，**不要清空/覆盖已有的 docs/data/**，保留上一次成功的数据（父母仍能看到上次的看板），同时走失败邮件流程。
7. **数值精度**：所有价格保留 2 位小数；成交量为整数（单位：手）。JSON 中数值用 number 类型，不要用字符串。
8. **时区**：运行日期以**北京时间（Asia/Shanghai）**为准。GitHub Actions 默认 UTC，须显式处理时区（见 7.2）。

## 6. 数据契约（★核心接口，DeepSeek 产出、Gemini 消费，双方严格遵守★）

> 所有 JSON 文件：UTF-8 编码、`ensure_ascii=false`（中文不转义）、数值为 number 类型、日期为 `"YYYY-MM-DD"` 字符串。

### 6.1 单只标的 K 线文件 `docs/data/kline/{code}.json`

```json
{
  "code": "600519",
  "name": "贵州茅台",
  "type": "stock",
  "adjust": "qfq",
  "dates":  ["2025-07-21", "2025-07-22", "..."],
  "kline":  [[1680.0, 1705.5, 1675.0, 1712.0], "..."],
  "volume": [123456, 98765, "..."],
  "ma5":    [null, null, null, null, 1699.8, "..."],
  "ma10":   [null, "...", 1702.3, "..."],
  "ma20":   [null, "..."],
  "ma60":   [null, "..."]
}
```

**关键约定（务必对齐）：**
- `dates`、`kline`、`volume`、`ma5/10/20/60` 这些数组**长度必须完全相等且一一对应**（同一下标是同一天）。
- `kline` 每个元素是 `[开盘, 收盘, 最低, 最高]` —— **注意这个顺序正是 ECharts candlestick 要求的 `[open, close, lowest, highest]` 顺序**，Gemini 可直接喂给 ECharts，无需重排。
- 均线在数据不足处填 `null`（不是 0，不是空字符串）。
- 数组按日期升序（老→新）。

### 6.2 当日摘要 `docs/data/summary.json`（首页列表用）

```json
{
  "items": [
    {
      "code": "600519",
      "name": "贵州茅台",
      "type": "stock",
      "last_close": 1705.5,
      "change_pct": 1.52,
      "change_amt": 25.5,
      "last_date": "2026-07-17",
      "status": "ok"
    }
  ]
}
```
字段说明：`last_close`=最新收盘价；`change_pct`=当日涨跌幅%（正为涨）；`change_amt`=当日涨跌额（元）；`last_date`=该只最新数据日期；`status`：`ok`=正常 / `stale`=用了旧数据 / `failed`=本次失败。
- `items` 顺序与 `watchlist.csv` 顺序一致，便于父母对照。
- 涨跌为正数表示上涨（前端红色），负数下跌（前端绿色），0 平盘（灰色）。

### 6.3 运行元信息 `docs/data/meta.json`（页面顶部状态条用）

```json
{
  "updated_at": "2026-07-17 18:32:10",
  "trade_date": "2026-07-17",
  "total": 15,
  "success": 15,
  "failed": 0,
  "failed_list": [],
  "run_status": "ok"
}
```
字段说明：`updated_at`=本次运行完成时刻（北京时间）；`trade_date`=本次数据对应交易日；`total`=自选标的总数；`success`/`failed`=本次成功/失败数；`failed_list`=失败代码列表（如 `["300750"]`）；`run_status`：`ok` / `partial`（部分失败）/ `failed`（全失败用旧数据）。

## 7. 自动化与失败通知（DeepSeek 负责，GitHub Actions）

### 7.1 触发时间
- 需求：**每交易日北京时间 18:30**。
- GitHub Actions cron 使用 **UTC**，北京时间 18:30 = UTC 10:30 → `cron: "30 10 * * 1-5"`（周一至周五）。
- 说明：cron 无法直接判断"是否交易日"，故周一至周五都触发；**遇到节假日（休市）当天没有新数据**，脚本应能识别"当天无新交易数据"这一情况：此时不报错、不发失败邮件，保留最近一个交易日的数据即可（把该情况当作正常）。
- GitHub Actions 定时任务存在数分钟到十几分钟的排队延迟，属正常现象，不影响使用。
- 同时保留 `workflow_dispatch`，允许维护者在网页上手动点一下立即运行（方便调试）。

### 7.2 workflow 关键点（`.github/workflows/daily.yml`）
- runner：`ubuntu-latest`，Python 3.11+。
- 步骤：checkout → 装依赖(`pip install -r requirements.txt`) → 运行 `python src/fetch_data.py` → 用 `git` 提交 `docs/data/` 变化并 push（从而触发 Pages 更新）。
- **时区处理**：在脚本内用 `zoneinfo`/`pytz` 明确取 `Asia/Shanghai` 时间来决定日期，**不要**依赖 runner 的系统时区。
- 提交数据需要写权限：在 workflow 中授予 `permissions: contents: write`，使用内置 `GITHUB_TOKEN` 提交即可。
- 提交信息示例：`chore(data): update dashboard data for 2026-07-17`。

### 7.3 失败邮件通知
- 触发条件：脚本以非零退出码结束（真正的失败：如全部标的抓取失败、依赖装不上、配置文件缺失），**或** `run_status == "failed"`。
- **不触发**的情况：节假日无新数据、个别标的失败但整体成功（属 `partial`，仅在页面标注，不发邮件，避免打扰）。
- 实现方式（二选一，DeepSeek 择简单者）：
  1. GitHub 账户设置里开启"仅在 Actions 运行失败时发邮件"（内置、零配置，最简单，**推荐**）——此时只要脚本失败退出，GitHub 自动发邮件给仓库所有者。
  2. 若需要更丰富内容，用一个发邮件的 Action（如 `dawidd6/action-send-mail`），配合 SMTP 密钥（存到仓库 Secrets）。
- 背书推荐用方案 1，最省事。README 中写明如何确认邮件开关已打开。

## 8. 前端可视化规格（Gemini 负责）

> **Gemini 注意**：你只需读取 `docs/data/` 下的 JSON（结构见第 6 章），用 ECharts 画图。**不要**去抓数据、不要算指标、不要引入构建工具。产物是纯静态 `docs/index.html` + `docs/assets/`，能被 GitHub Pages 直接托管。

### 8.1 技术要求
- 纯静态：`index.html` + `app.js` + `style.css`，无需 npm 构建。
- 图表库：**ECharts 5.x**，用官方 CDN 引入（如 jsDelivr）。可加一个国内可用的备用 CDN 兜底。
- 移动端优先：父母主要用手机竖屏看，必须响应式；图表宽度自适应，触摸可缩放/平移 K 线（ECharts dataZoom）。
- 中文与字体：确保中文正常显示，正文字号偏大（考虑中老年用户，正文 ≥ 16px，标题更大）。

### 8.2 颜色规范（A股习惯，务必遵守）
- **涨 = 红色，跌 = 绿色，平 = 灰色**（与 A 股软件一致，切勿用美股的绿涨红跌）。
- K 线：阳线（收≥开）红色，阴线（收<开）绿色。
- 建议主色调清爽明亮，背景浅色（中老年用户在户外/明亮环境看手机更清楚）；避免低对比度的灰字。
- 成交量柱：颜色跟随当日涨跌（红/绿）。

### 8.3 页面结构（三部分）

**① 顶部状态条**（读 `meta.json`）
- 显示"数据更新时间：2026-07-17 18:32（交易日 2026-07-17）"。
- 若 `run_status = partial`，温和提示"部分标的今日更新失败"；若 `failed`，提示"今日更新失败，当前显示为上次数据"。用户能一眼知道数据新不新。

**② 自选列表概览**（读 `summary.json`）
- 卡片或列表形式，每只一行/一卡：名称、代码、最新价、涨跌幅（红/绿，带 ↑↓ 或 +/-）、涨跌额。
- 涨跌幅要醒目（大字、红绿）。这是父母最关心的"今天涨了还是跌了"。
- 点击某只 → 展开/跳转到该只的 K 线图（③）。
- `status=stale/failed` 的标的做灰色或小标记提示。

**③ K 线详情**（读 `kline/{code}.json`）
- 主图：日 K 线蜡烛图 + MA5/MA10/MA20/MA60 均线（不同颜色，配图例，可点击图例显隐）。
- 副图：成交量柱状图，与主图 X 轴联动。
- 交互：dataZoom 支持缩放/平移；tooltip 显示某日的 开/高/低/收/涨跌幅/成交量/各均线值。
- 默认展示最近 1 年（约 250 个交易日）；数据里可能多带了一些，前端截取最近 1 年展示即可。
- 标题显示 名称 + 代码 + 最新价 + 涨跌幅。

### 8.4 可读性与容错
- 首屏 3 秒内出图；加载中显示"加载中…"。
- 某只 JSON 读取失败时，该只显示"暂无数据"，不要整页崩。
- 一切文案用中文，避免专业黑话（面向不熟悉技术的父母）。

## 10. 面向维护者（子女）的配置步骤（README 应包含，此处给要点）

DeepSeek 写 README 时，请把下列步骤写成中老年子女也能照做的图文级教程：

1. 注册 GitHub 账号（免费）。
2. 新建一个仓库（建议名 `stock-dashboard`，可设为 Public 以便免费用 Pages；若设 Private 需确认账户 Pages 权限）。
3. 把本项目文件传上去（网页上传或 git）。
4. 在仓库 **Settings → Pages** 里，Source 选择 **Deploy from a branch**，分支选 `main`、目录选 `/docs`，保存后得到公网网址。
5. 在仓库 **Settings → Actions → General** 里，确认 Workflow 有读写权限（`Read and write permissions`），以便自动提交数据。
6. 在 GitHub **个人 Settings → Notifications** 里，确认"Actions"失败通知邮件已开启。
7. 打开 **Actions** 标签页，手动点一次 `Run workflow` 测试，成功后访问 Pages 网址验证。
8. 把网址发给父母，教他们**存成手机浏览器书签 / 添加到主屏幕**，以后点一下就看。
9. 以后要增删自选股，只需在网页上编辑 `watchlist.csv` 提交即可，次日自动生效。
